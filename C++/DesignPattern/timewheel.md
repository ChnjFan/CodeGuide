# 时间轮设计

高性能场景下，后台处理大量延时任务怎么设计？

首先要明确业务场景中的任务规模和时间精度要求，合理选择数据结构和触发延时任务机制。

如果任务规模不大可以使用最小堆或者红黑树实现，如果任务规模几十万甚至上百万需要使用时间轮。

1. 数据结构的选择：组织所有的延时任务。
2. 触发延时任务：如何触发最近要执行的任务。

## 数据结构设计

常用的定时器数据结构有最小堆、红黑树（适合规模小的任务）和时间轮。

高性能场景需要考虑：

1. 多线程并发
2. 内存分配（对象池）和缓存局部性
3. 定时精度和吞吐量权衡（触发时间）
4. 性能压测指标

### 时间轮原理和机制

**时间轮（Timing Wheel）**是一种高效的延时 / 定时任务调度算法，用**环形数组 + 指针 + 任务链表**模拟时钟，把任务按延迟映射到 “时间槽”，指针定时推进并批量触发到期任务，整体**O(1)** 增删与调度，适合海量超时、重试、心跳类任务。

#### 单层时间轮结构

环形数组：固定数量的槽位，每格代表时间粒度。

指针按照粒度定时前进一格，指向当前的时间槽，每个槽位中挂载一个链表存储到期要执行的任务。

例如粒度为 100ms 一共 512 个槽位的时间轮，任务延时 2.3s （23 格），如果当前指针在第 5 格，那么任务应该放在 28 槽，圈数 round 为 0。

对于长任务通过每次指针遍历到对应槽位时，round - 1 == 0 时再执行任务。例如延时 60s 任务，任务应该放在 600 % 512 = 1 圈余 88 格，放在 88 槽 round = 1。指针到 88 槽时 round - 1，直到 round 减到 0 时下一圈就执行任务。

#### 多层时间轮结构

多层时间轮有多个环形数组，每个环形数组对应自己的时间跨度。

例如可以分为秒轮、分轮和时轮，在高层任务到期后，将任务下沉到底层继续等待，最终由底层触发执行。

#### 时间轮推进

指针移动有两种方法：

1. sleep 空推进：sleep 会调用系统调用，存在系统调用和上下文切换的开销。
2. 通过 condition_variable::wait_until 按照最低层级或映射的时间点阻塞时间轮线程，唤醒后指针偏移到对应存在任务的槽通知业务线程执行任务。

## 触发延时任务

时间轮由单独的线程处理。

1. 业务线程添加延时任务后，由时间轮线程负责记时和到期触发，避免执行延时任务阻塞时间轮线程影响其他任务执行。
2. 任务到期后时间轮触发业务线程执行延时任务。

任务到期后时间轮线程通过阻塞队列通知任务线程执行延时任务。

## 优化方案

任务创建使用对象池来优化内存申请释放的开销。

时间轮添加延时任务可以使用无锁队列减少锁的开销。

多个时间轮线程，通过对工作线程进行哈希分片来均衡每个时间轮延时任务的负载。

## 性能指标

正确性：

1. delay 小于等于 0 时是否立即执行任务
2. 普通的短延时任务是否能触发
3. 跨多层时间轮的延时任务是否能触发
4. 取消任务是否还会执行
5. 延时任务回调是否在业务线程执行
6. 析构时后台线程能否正常退出

压测指标：

1. add QPS 每秒添加定时器数量
2. cancel QPS 每秒取消定时器数量
3. expire QPS 每秒到期任务数量
4. 触发延时：实际执行时间 - 期望执行时间
5. CPU 利用率和内存占用
6. 锁竞争：多个工作线程并发添加延时任务的性能瓶颈
