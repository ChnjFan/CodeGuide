## 面试问题

### 为什么选择MQTT协议作为命令通信协议

我们项目面向**卫星激光通信场景**，卫星链路**带宽资源有限、丢包率高、链路容易中断、往返时延大**，业务上需要实现远程命令执行、设备状态采集、告警批量推送，要求协议兼顾**低带宽占用、传输可靠、一对多分发**能力。

> 需求背景：卫星激光通信终端需要高效、可靠和轻量的管理能力，在 10Mbps 带宽，5% 随机丢包，RTT 最大 600ms 左右的网络环境下稳定维持管理会话，实现命令行管理维护终端。

我们依次评估了三类备选方案。

- **SNMP**：经典网管协议，基于 UDP 无连接传输，不具备消息重传、有序交付能力，在高丢包链路中数据可靠性无法保障，直接排除。
- **Netconf**：依托 XML 承载配置与数据，文本格式冗余度高，报文载荷大，在窄带宽链路中传输效率极低，不适用。
- **gRPC**：序列化采用 Protobuf，压缩率高，但底层基于 HTTP/2，协议栈厚重、建链握手步骤多。卫星链路频繁闪断时，重连成本高，长连接稳定性差，无法满足长期稳定通信的要求。

MQTT 作为专为**低带宽、不稳定网络**设计的物联网通信协议，各项特性高度匹配：

- 协议设计极简，最小报文头仅 2 字节，报文整体体积小，最大化利用卫星有限带宽；
- 支持 QoS0/1/2 三级服务质量，搭配消息重传、心跳保活机制，有效对抗网络丢包与链路抖动，保证指令、告警数据可靠投递；
- 发布订阅的通信模型，解耦通信双方，原生支持一对多广播，完美适配网管集群管控海量设备、告警多节点同步的业务架构。

综合网络环境、传输性能、业务架构三方面因素，MQTT 是当前场景下最优选择。

### MQTT 命令行通道的整体架构

我们设计的整体通信架构是在地面部署 MQTT Broker，星载设备作为 MQTT 客户端。

地面 Broker 负责消息路由、会话保持、离线缓存与消息分发；设备作为 MQTT 客户端，对接原有 CLI 组件处理命令执行与告警、状态数据上报。

传输上用 protobuf 序列化传输数据降低带宽占用，采用 QoS1 保证命令、告警数据可靠送达，配合离线消息和会话保持解决链路中断的消息发送。Topic 按照命令请求和应答，告警状态数据来设计。

网管单个消息发送多条配置命令执行，设备侧默认在手动模式执行后提交来保证命令执行一致性，存在一条命令执行失败或者最后提交失败，都会返回执行该消息的全部命令执行失败，不会存在一部分执行成功，一部分执行失败的情况。

### 详细讲解 MQTT 协议怎么保证可靠传输数据

通过 MQTT 协议的持久会话、离线消息补发机制、QoS 消息可靠性和重连机制保证消息可靠性。

#### 会话保持怎么设计

首先设备侧在连接时设置 clean session 为 false 开启持久会话，由地面的 Broker 保存会话订阅关系、未确认消息和离线期间的指令，终端重连自动恢复会话，补发消息，不用重新订阅，重新建链。`cleanstart = 0`：**恢复旧会话**，复用之前持久化的会话状态，保留订阅、离线消息等，实现断点续会话。

> 开启持久会话之后要保证 **客户端 ID** 唯一不变，服务端通过客户端ID+服务端地址识别客户端是否存在历史会话。

`keepAliveInterval` 保活时间设置为 180 s 确保 Broker 不会频繁误判链接断开频繁重连，造成消息无法送达。

> 设置 keepAliveInterval  后 paho 会按照这个时间间隔来发送 PINGREQ 消息保活，如果 Broker 在 1.5 倍时间内没有收到就断开连接。

星载设备终端设置会话过期时间 `SESSION_EXPIRY_INTERVAL` 默认为 3 小时，覆盖卫星短时间离线，超时自动回收会话资源。用户可将核心骨干节点配置永久保留或自定义时间。

#### 离线消息怎么设计

对数据进行严格的 QoS策略分级。

对于命令下发和告警状态的上报，采用 QoS1 级别发送消息，保证消息不会丢失。对于心跳数据或者日志，采用 QoS0 发送消息来节省带宽。

设备离线或链路中断时地面 Broker 自动缓存消息，重连后补发缓存的离线消息。

> 为什么不用 QoS2？ 因为卫星环境的时延高，QoS2 都需要 4 次交互，导致时延翻倍，如果发生重传的话，链路中会有更多的消息占用带宽

reliable 设置为 true，保证消息有序发送，简单可控，避免同时发送一堆消息造成超时重传风暴。

> 1. 设为`true`（可靠模式）：一条消息收到确认回执后，才能发送下一条；多发消息会返回`MQTTCLIENT_MAX_MESSAGES_INFLIGHT`错误码。
> 2. 设为`false`：最多允许**10 条**消息同时处于待确认状态，特定场景下可提升数据传输吞吐量。

设置遗嘱消息并设置成保留消息，网管能快速感知到设备离线停止发送指令。

`retryInterval`  会话内超时重传时间根据官方文档的描述说明设置为 0 ，依靠 TCP 的重传机制保证消息的可靠性。

业务层每条消息都会携带消息序列号，接收方收到后会回复命令执行结果并携带消息序列号，根据序列号做消息过滤，防止链路重连导致的消息重复接收。

### 传输数据使用protobuf如何设计数据帧结构

我采用**4 字节固定头部 + Protobuf 单入口消息结构**。头部用于快速路由、判断序列号幂等、版本兼容；

业务数据使用 protobuf 设计，使用 oneof 保存指令内容、告警信息或状态数据。

```
┌─────────────────────┬───────────────────────────┐
│    固定头部(4字节)    │     Protobuf 序列化载荷     │
├────────┬────────┬───┼───────────────────────────┤
│ 版本号  │ 消息类型 │ 序列号│  业务数据（protobuf）  │
│  1B    │   1B   │  2B  │     变长二进制          │
└────────┴────────┴───┴───────────────────────────┘
```

数据兼容性依靠 protobuf 的标准规范保证新旧终端和新旧网管的互通：

- 字段号分配后不修改、不复用。
- 新增字段需要用 optional 或者设置保留默认值，废弃字段标记 reserved 不删除。
- 通过版本号来进行兼容判断。

扩展性设计：

- 通过 oneof 来管理不同类型消息，所有消息类型都在一个 Message 结构中。
- 消息头中预留 4 字节拓展字段给未来使用。
- 通过消息头中的序列号来保证消息不会重复接收。

### 采用 Protobuf 后，带宽占用相比原有方案降低了多少，有量化数据吗？

相比于 JSON 报文单条报文，Protobuf 是二进制 TLV 紧凑编码，不用段名、无引号 / 逗号等冗余，配合 Varint 压缩，比 JSON 文本格式小非常多。

### 如何设计 MQTT 通道与 CLI 组件适配

将 MQTT 的 paho 协议库嵌入到接入协议组件中，在 CLI 组件中增加中间适配层负责 MQTT 消息接收与解析，将 MQTT 消息转换为 CLI 标准输入消息，CLI 内部增加 MQTT 终端会话专门执行 MQTT 协议发送的命令。

CLI 组件接收到命令执行结果，告警和 Telemetry 数据后通过 MQTT 适配层进行消息封装，回复给 Broker。

### MQTT 命令行通道的并发控制策略是什么，如何支持多用户、多命令并发执行？

设备侧命令通过串行执行，MQTT 适配层接收到的全部命令消息都通过队列进行缓存。

命令队列中设置消息优先级，产品定制关键指令优先级，例如重启命令优先执行，普通的查询命令可以设置为低优先级排队等待。

命令执行超时由 CLI 组件的命令执行超时机制支持，MQTT 适配层能获取到命令执行结果返回给网管。

队列消息如果超过阈值后会拒绝接收新的操作，主动向网管发送消息反压，网管侧停止继续发送消息。队列超过阈值后只能接受最高优先级的关键指令（如重启操作），保证设备能及时复位。

### 项目中如何对 MQTT 命令行通道进行性能测试，关键指标有哪些？

我们在开发完成后，在 PC 仿真上使用 Linux 命令 `tc` 设置时延和随机丢包模拟卫星环境，进行了功能基准测试、压力测试和可靠性测试。

```bash
# 600ms 时延 + ±200ms 抖动，丢包率5%（更贴近真实卫星链路）
tc qdisc add dev eth0 root netem delay 600ms 200ms loss 5%
```

关键性能指标：

- 消息可靠性：模拟环境下消息可靠传输无丢失。
- 带宽占用：通过 `tcpdump` 抓包统计消息长度，计算命令执行期间平均带宽小于 500bps。
- 离线消息能力：测试重连补发消息全部送达，无丢失。
- 稳定性指标：运行无内存泄露，无命令阻塞，程序无异常崩溃。

## 基础概念

### 简述 MQTT 协议的基本架构、组成角色（客户端、服务端 / 代理、发布者、订阅者）。

MQTT 采用**发布 - 订阅（Pub/Sub）** 架构，所有消息流转都经过中间代理节点，解耦发布方与订阅方：双方无需同时在线、无需互相知晓地址，仅通过**主题**完成消息路由，天然支持一对多、多对多分发，适配你项目中设备状态采集、告警批量推送场景。

服务端（MQTT Broker）整个系统的核心枢纽，主要职责：

- 维护全网客户端连接、心跳、会话信息；
- 接收客户端发布的消息，基于主题做路由转发；
- 支持离线消息缓存（配合 QoS 与会话），断链恢复后补发；
- 管理主题、订阅关系，实现消息的一对多分发。

使用 MQTT 协议接入 Broker 的终端 / 应用，统一称为**客户端**。**一个客户端可同时兼具发布者和订阅者身份**。

发布者（Publisher）负责主动向 Broker 的指定 **Topic** 发布业务消息，比如卫星设备上报状态、推送告警、接收端下发远程命令。

订阅者（Subscriber）主动向 Broker 订阅目标主题，Broker 收到对应主题消息后，会主动推送给该客户端，比如地面平台采集设备数据、接收告警通知。

**标准工作流程**

1. 所有客户端先和 Broker 建立 TCP 连接，完成 MQTT 握手；
2. 订阅者提前订阅需要监听的主题；
3. 发布者向对应主题发布消息；
4. Broker 接收消息，检索所有订阅该主题的客户端；
5. Broker 将消息转发至全部匹配的订阅者，完成通信。

**架构优势**非常适配项目：

1. 解耦设计，适配卫星链路**频繁断连、往返时延大**的特点，客户端不用永久在线；
2. 天然支持**一对多分发**，完美匹配告警批量推送、多设备状态采集需求；
3. Broker 支持会话保留、离线消息缓存，可在链路中断恢复后补传数据，提升弱网可靠性。

### MQTT 基于什么传输层协议？默认端口是多少？

标准 MQTT 协议依托 **TCP** 实现，面向连接、可靠传输，依托 TCP 完成握手、重传、流量控制；

- 未加密明文通信：默认端口 **1883**；
- 基于 TLS/SSL 加密的安全通信：默认端口 **8883**；

### 解释 MQTT 主题（Topic）、主题层级、通配符（+、#）的用法和使用限制。

主题（Topic）是 MQTT 中消息的唯一路由标识，本质是 UTF-8 字符串，发布和订阅依靠主题匹配实现消息转发。

主题层级使用正斜杠 `/` 作为分隔符，将主题划分为多级层级，结构类似文件路径。协议允许空层级、首尾带 `/`，但属于不规范用法，生产环境不推荐。

**通配符用法（仅订阅可用）**

单层通配符 `+` **匹配任意单个层级**，占位且只占一层。

多层通配符 `#` **匹配当前层级及往下所有子层级**，支持 0 层、多层任意内容。**必须位于主题最后一位**，且前方必须搭配 `/`。

### MQTT 三种 QoS 等级分别是什么？各自的工作流程、可靠性、适用场景？

QoS 是**发布者 → Broker、Broker → 订阅者**两段链路分别独立保障；等级协商以**双方最低等级**为准。弱网 / 长时延场景下，高 QoS 会显著增加时延与带宽消耗。

#### QoS 0：At Most Once 最多一次

**工作流程**

1. 发布方直接向 Broker 发送 **PUBLISH** 报文；
2. 发送完成后立即清空本地缓存，**无需等待应答、不重传**。

**可靠性**

- 最低等级，**消息可能丢失**，不会重复；（主要是在 Broker 缓冲区或客户端缓冲区满时丢失）
- 链路断连、丢包时，数据直接丢失。

**适用场景**

- 优点：流程最简单、报文最少、带宽 / 时延开销最小；
- 适用：数据实时性优先、允许少量丢包的**非关键高频数据**。
- 结合你的场景：卫星常规设备状态、周期性采样数据。

#### QoS 1：At Least Once 至少一次

**工作流程（两次握手）**

1. 发布方发 **PUBLISH** 报文，本地缓存报文并启动超时计时器；
2. Broker 收到后，回复 **PUBACK** 应答报文；
3. 发布方收到 PUBACK，删除本地缓存，流程结束；
4. 超时未收到应答，**重复重发 PUBLISH**，直到收到应答。

**可靠性**：保证消息**一定能送达**，但因重传机制，**订阅方会收到重复消息**。

**适用场景**

- 开销中等，是业务最常用等级；
- 适用：必须送达、业务可做**去重处理**的消息。
- 结合你的场景：卫星告警推送、普通远程控制指令。

#### QoS 2：Exactly Once 恰好一次

**工作流程（四次握手，全程防重、防丢）**

1. 发布方发送 **PUBLISH**，缓存报文，等待响应；
2. Broker 收到后回复 **PUBREC**（已接收）；
3. 发布方收到 PUBREC，删除原 PUBLISH 缓存，发送 **PUBREL**（释放报文）；
4. Broker 收到 PUBREL，完成消息转发，回复 **PUBCOMP**（完成）；
5. 发布方收到 PUBCOMP，流程结束。任意环节超时，对应报文都会重传。

**可靠性**：最高等级，**既不丢失、也不重复**，实现精准一次投递。

**适用场景**

- 交互报文最多、流程最长，**时延和带宽开销最大**；长时延 + 高丢包的卫星链路中，性能损耗尤为明显。
- 适用：对重复消息零容忍、核心关键业务。
- 结合你的场景：卫星参数配置、设备启停、关键执行指令等。

### 什么是 MQTT 遗嘱消息（Will Message）？作用是什么？

遗嘱消息是客户端**建立连接时**预先在 Broker 配置的一条消息。当客户端**异常断开连接**（非主动正常下线）时，Broker 会自动向指定主题发布这条遗嘱消息。主要用于**在线状态监测、异常断链告警、设备离线通知**。

客户端在发起 MQTT **CONNECT 连接请求**时，可携带遗嘱配置：包含**遗嘱主题、遗嘱内容、QoS、保留标识**。

客户端**非正常断开**时触发：TCP 链路异常中断、网络闪断、设备断电、程序崩溃、心跳超时等。客户端主动发送 **DISCONNECT** 报文正常下线，**不会**发布遗嘱消息。

卫星链路频繁闪断、设备在轨无法现场运维，遗嘱消息无需轮询心跳，链路异常时自动推送离线通知；区分**主动下线**和**意外断链**，精准识别是设备故障还是卫星链路问题。

- 遗嘱消息**必须在建连时配置**，连接建立后无法动态修改；
- 遗嘱消息也支持 QoS 0/1/2 和 Retain 保留消息标识；
- 心跳（Keep Alive）超时是触发遗嘱消息最常见的条件。

### 什么是保留消息（Retained Message）？使用场景和注意事项？

保留消息是发布消息时开启 **Retain 标识** 的特殊消息。Broker 会为对应主题**永久存储最新一条保留消息**。

新客户端订阅该主题时，Broker 会立即推送这条最新保留消息。常用于下发设备配置、同步初始状态、广播固定参数；使用时注意清理旧消息、控制数量，避免占用资源。

客户端发布消息时，将报文的 **Retain 位置为 1**，该消息即为**保留消息**。

- Broker 收到后，会**覆盖并存储该主题下最新的一条保留消息**，旧保留消息直接作废；
- 后续有客户端**新订阅此主题**，Broker 会立刻把这条保留消息推送给它，无需等待新的发布动作；
- 普通消息仅推送给**当前已在线订阅**的客户端，保留消息可触达**后续新增订阅**的客户端。

**清除保留消息的方式**

- 向同一主题发布一条 **Retain=1、载荷为空** 的消息，Broker 会删除该主题的保留消息；
- 部分 Broker 支持后台手动清理、配置过期时间。

**核心注意事项**

- 一个主题永远只保留**最后一条**保留消息，新的保留消息会直接覆盖旧内容。
- Retain 只对**新订阅者**生效，已在线的订阅者正常接收消息，不会重复推送历史保留消息。
- 长期不用的保留消息会一直占用 Broker 内存，尤其卫星场景终端多、主题多，堆积过多会消耗资源。
- 保留消息会继承发布时的 QoS 等级，关键配置建议用 QoS1 保证送达。
- 像实时温度、瞬时告警这类频繁变化的数据，不适合用保留消息，会频繁覆盖且无意义。
- 保留消息存在于 Broker 全局，和客户端会话是否持久无关。

### MQTT 心跳包（Keep Alive）的作用、工作机制是什么？

MQTT 心跳保活（Keep Alive）用于**检测连接有效性**，防止客户端静默离线、服务端无效占用连接。

客户端建连时指定心跳时长，周期发送心跳报文；若服务端在**1.5 倍心跳时间**内未收到任何数据 / 心跳，判定客户端掉线，主动断开连接并触发遗嘱消息。

**核心作用**：

- **检测死连接**：识别客户端异常断电、网络断连、程序卡死等**静默离线**（无正常 DISCONNECT 报文），释放 Broker 连接资源。
- **维持网络链路**：防止中间网关、防火墙因长时间无数据交互，主动断开空闲 TCP 连接。
- **遗嘱消息**：心跳超时判定离线后，Broker 会触发客户端预设的**遗嘱消息**。

客户端在 `CONNECT` 报文中指定 **Keep Alive 时长**（单位：秒），代表最大空闲间隔。空闲期间，客户端需**周期性**向 Broker 发送 **PINGREQ**（心跳请求）；Broker 收到后立即回复 **PINGRESP**（心跳响应），完成一次心跳交互。若客户端本身有业务消息收发，**可替代心跳**，无需额外发 PINGREQ。

心跳时长设为 **0**：代表**关闭心跳检测**，Broker 不会主动因空闲断开连接。心跳由**客户端主动发起**，服务端不会主动探测。

卫星链路**时延大、易闪断、带宽宝贵**，心跳配置有特殊考量：

1. 不宜设置过小的心跳周期：频繁心跳会额外消耗窄带宽，长时延也容易造成**误判离线**；
2. 心跳周期建议适当放大，适配卫星大往返时延；
3. 心跳超时精准识别链路 / 设备异常，配合遗嘱消息实现自动离线告警，适配在轨无人运维场景。

### MQTT 会话（Session）的概念，持久会话和非持久会话的区别？

MQTT 会话是 Broker 为客户端维护的**连接状态与数据集合**。通过建连时 `Clean Session` 字段区分两种模式：

1. **非持久会话（Clean Session = true）**：连接断开后，会话立即清空，离线消息、订阅关系全部丢弃；
2. **持久会话（Clean Session = false）**：断连后会话保留，Broker 缓存离线消息、留存订阅，重连后恢复状态并补发消息。

客户端与 Broker 建立 MQTT 连接后，Broker 会创建专属**会话**，用于记录该客户端的运行状态，主要包含三类数据：

- 客户端已订阅的**主题列表**；
- 客户端离线期间，待推送的 **QoS 1 / QoS 2 离线消息**；
- QoS 2 消息交互过程中的**半完成状态数据**。

会话生命周期由 `Clean Session` 控制，该字段在 **CONNECT 建连报文** 中指定。

持久会话有**过期时间**，超时后仍未重连，Broker 会自动清理会话与缓存消息；仅对 QoS1/QoS2 生效，QoS0 消息永远不做离线缓存。

卫星链路易闪断、设备常离线重连、在轨无人值守：

1. 采集状态、普通高频数据：可用**非持久会话**，减少 Broker 缓存压力；
2. 告警、远程指令等关键 QoS1/QoS2 消息：必须开启**持久会话**，避免断链丢失重要数据；
3. 需合理配置会话过期时间，防止大量离线会话长期占用服务端资源。

> 开启持久会话还消息可能的原因：QoS 0 消息不缓存；会话已经过期；Broker 中的离线消息队列满；客户端重连时变更了 ClientID。

## 协议适配类

### MQTT 原生协议哪些地方会造成带宽浪费？如何优化 MQTT 以降低带宽占用？

通过抓包发现 MQTT 报文头部开销主要在 topic 上，通过缩短主题名、减少层级来缩短头部大小。固定主题前缀，用编号替代长字符。通过使用工业级的 Broker 实现主题别名，通过短主题来映射到长主题上，减少卫星终端到 Broker 链路上的报文大小。

消息重传间隔应该设置为 0，避免在长 RTT 的链路，消息超时造成大量的重传消息在链路上。

不要使用 QoS 2 的消息，应该在业务层自行做消息去重。QoS 2 的消息链路上需要传输 4 次报文完成消息投递。

启用持久会话，断链重连后**无需重新订阅主题**，大幅减少 CONNECT、SUBSCRIBE 等控制报文。

### MQTT 报文结构包含哪些字段？哪些字段可以精简、压缩来减少报文体积？

MQTT 报文由**固定头部、可变头部、有效载荷**三部分组成。固定头部为必选，可变头部、载荷按需存在。

可优化压缩的部分：**剩余长度、主题字符串、报文标识符、协议版本 / 标志位、载荷数据**；

固定头部首字节、控制报文类型与标志位属于协议强制字段，**无法删减**。

- 字节 1：报文控制字段（8 位）**高 4 位**：**报文类型**（PUBLISH、CONNECT、SUBSCRIBE、PINGREQ 等，共 14 种），**高 4 位**：**报文类型**（PUBLISH、CONNECT、SUBSCRIBE、PINGREQ 等，共 14 种）
- 字节 2~N：**剩余长度（Remaining Length）**表示「可变头部 + 载荷」的总字节数，采用**可变长度编码**（1~4 字节）。

可变头部（Variable Header，按需存在，可大量精简）

- **协议名 + 协议版本**（仅 CONNECT 连接报文）固定字符串 `MQTT` + 版本号，建连时必传；频繁重连场景会反复传输。
- **报文标识符 Packet ID**（报文 ID，2 字节）：仅 **QoS1、QoS2、SUBSCRIBE、UNSUBSCRIBE** 报文需要匹配请求与应答。
- **主题名称 Topic**（PUBLISH 报文核心字段，字符串）
  - 问题：长字符串主题反复传输，开销极大；
  - 优化：主题缩写、层级简化、配置主题别名。
- **遗嘱、心跳、保留消息相关标识**（CONNECT 报文）多余配置字段会增加可变头部长度。

### QoS 1、QoS 2 在高丢包 + 长时延链路下，会出现什么问题？如何解决？

高丢包 + 长时延环境中，QoS1/QoS2 会出现**消息阻塞、频繁超时重传、往返时延叠加、带宽被应答报文挤占、重复消息增多**等问题，QoS2 问题更严重。

解决思路：优先降级使用 QoS0 + 业务层补传、QoS1 配合业务去重；调优重传策略、拆分报文、消息聚合、客户端本地缓存；必要时简化交互逻辑。

高丢包长时延下，QoS1 易阻塞、重传多、重复消息多；QoS2 时延与开销翻倍、故障点更多。优先降级 QoS、业务层兜底去重，再调优超时重传、做消息聚合与本地缓存，尽量少用 QoS2。

### 卫星通信往返时延大，会对 MQTT 的握手、消息应答、心跳机制产生哪些影响？

卫星大往返时延 RTT 拉长了 MQTT 所有交互报文的等待窗口，极易出现**超时误断连、QoS 频繁不必要重传、握手失败、心跳误踢下线**三类问题，必须针对性调参适配长时延。

卫星链路 RTT 可达数秒，远大于地面网络，三方面影响：

1. **握手**：CONNECT-CONNACK 往返耗时变长，默认建链超时容易建连失败、频繁重连；优化放大建链超时、指数退避重连。
2. **QoS 应答**：应答报文跨链路耗时久，默认超时容易误判丢包、多余重传；采用 retryInterval=0 关闭在线重传、上调应答超时、限制 inflight 报文。
3. **心跳**：短心跳参数下 PINGRESP 未传回就被服务端踢下线；将 KeepAlive 放大至 120~300s，大于 3 倍最大 RTT，避免误断连。

### 长 RTT 下，QoS 交互流程（握手机制）耗时变长，会引发什么问题？如何优化？

QoS1 一次往返、QoS2 四次往返全依赖 RTT，链路时延拉满后，未确认报文堆积、inflight 打满、误超时重传、发送阻塞是四大核心问题，围绕「控并发、停在线重传、调超时、生命周期管控」优化。

1. inflight（飞行窗口）快速打满，后续消息发送阻塞。reliable 设置为 true，一条消息发送成功后再才能发送下一条消息，避免窗口瞬间占死、发送阻塞；
2. 应答超时时延误判丢包，触发大量无效重传。遵循 paho 官方建议，**链路正常时不再定时重传未 ACK 报文**，彻底杜绝因大 RTT 超时误重发；仅在断链重连后依靠持久会话统一补发。
3. QoS 级别越高延时放大越高，关键命令使用 QoS 1 级别通过业务层去重保证可靠性，避免使用 QoS 2 消息。

### 客户端连接 Broker 的超时时间、消息应答超时时间如何调优？

在卫星长 RTT 场景下，连接超时和消息应答超时必须按**链路最大 RTT 的倍数**放大：

1. **连接超时**设置为**3 倍 RTT 以上**，推荐 20~30 秒，保证 TCP+MQTT 握手能完成，不出现建连失败；
2. **消息应答超时**设置为**2 倍 RTT 以上**，推荐 15~25 秒，确保应答回来前不触发超时，避免无效重传；
3. 同时必须配置 **retryInterval=0** 关闭在线重传，只依靠持久会话重连补发。

### 长时延场景下，大量并发消息堆积如何处理？会不会造成 Broker 或客户端阻塞？

在卫星长时延场景下，**大量并发消息一定会造成堆积，并且会阻塞客户端和 Broker**。

因为 QoS 消息需要等待 ACK，而卫星 RTT 大、应答慢，导致 inflight 飞行窗口迅速占满，消息只能排队，队列膨胀最终造成客户端阻塞；大量设备同时堆积还会压垮 Broker。

处理方案有五点：

1. **缩小 inflight 窗口到 5~10**，从源头限制并发；
2. **非关键数据改用 QoS0**，不占用确认资源；
3. **客户端做发送频率限流**，避免瞬间爆发；
4. **设置队列上限和溢出策略**，防止内存打满；
5. 断连重连后**分批限速补发**，避免流量风暴。
