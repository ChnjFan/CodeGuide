### 为什么选择MQTT协议作为命令通信协议

我们项目面向**卫星通信场景**，卫星链路**带宽资源有限、丢包率高、链路容易中断、往返时延大**，业务上需要实现远程命令执行、设备状态采集、告警批量推送，要求协议兼顾**低带宽占用、传输可靠、一对多分发**能力。

> 卫星弱网主要是网络慢+容易丢包

我们依次评估了三类备选方案。

- **SNMP**：经典网管协议，基于 UDP 无连接传输，不具备消息重传、有序交付能力，在高丢包链路中数据可靠性无法保障，直接排除。
- **Netconf**：依托 XML 承载配置与数据，文本格式冗余度高，报文载荷大，在窄带宽链路中传输效率极低，不适用。
- **gRPC**：序列化采用 Protobuf，压缩率高，但底层基于 HTTP/2，协议栈厚重、建链握手步骤多。卫星链路频繁闪断时，重连成本高，长连接稳定性差，无法满足长期稳定通信的要求。

MQTT 作为专为**低带宽、不稳定网络**设计的物联网通信协议，各项特性高度匹配：

- 协议设计极简，最小报文头仅 2 字节，报文整体体积小，最大化利用卫星有限带宽；
- 支持 QoS 0/1/2 三级服务质量，搭配消息重传、心跳保活机制，有效对抗网络丢包与链路抖动，保证指令、告警数据可靠投递；
- 发布订阅的通信模型，解耦通信双方，原生支持一对多广播，完美适配网管集群管控海量设备、告警多节点同步的业务架构。

综合网络环境、传输性能、业务架构三方面因素，MQTT 是当前场景下最优选择。

### 详细讲解 MQTT 协议怎么保证可靠传输数据

首先整体通信架构是将 Broker 放在地面，卫星终端与 Broker 之间通过卫星链路弱网环境连接，网管与 Broker 之间做集中管控和指令下发。

Broker 充当缓冲中心，所有指令先到达 broker，然后上报卫星。卫星测设备只保证最终可达，不要求实时性，网管侧保证实时可控，状态可查，可追溯。可靠性全部由终端与 broker 之间来保证网管不需要直接面对弱网环境。

首先，对数据进行严格的 qos 策略分级。

对于管控指令下发和告警状态的上报，采用 qos 1 级别发送消息，保证消息不会丢失。对于心跳数据或者日志，采用 qos 0 发送消息来节省带宽。

> 为什么不用 qos 2？ 因为卫星环境的时延高，qos 2 都需要 4 次交互，导致时延翻倍，如果发生重传的话，链路中会有更多的消息占用带宽。

设备侧在连接时设置 clean session 为 false 开启持久会话。由 Broker 保存订阅关系、未确认消息和离线期间的指令，终端重连自动恢复会话，补发消息，不用重新订阅，重新建链。`cleanstart = 0`：**恢复旧会话**，复用之前持久化的会话状态，保留订阅、离线消息等，实现断点续会话。

> 开启持久会话之后要保证 **客户端 ID** 唯一不变，服务端通过客户端ID+服务端地址识别客户端是否存在历史会话。

keepAliveInterval 保活时间设置为 180 s 确保 Broker 不会频繁误判链接断开频繁重连，造成消息无法送达。

> 设置 keepAliveInterval  后 paho 会按照这个时间间隔来发送 PINGREQ 消息保活，如果 Broker 在 1.5 倍时间内没有收到就断开连接。

reliable 设置为 true，保证消息有序发送，简单可控，避免同时发送一堆消息造成超时重传风暴。

> 1. 设为`true`（可靠模式）：一条消息收到确认回执后，才能发送下一条；多发消息会返回`MQTTCLIENT_MAX_MESSAGES_INFLIGHT`错误码。
> 2. 设为`false`：最多允许**10 条**消息同时处于待确认状态，特定场景下可提升数据传输吞吐量。

设置遗嘱消息并设置成保留消息，网管能快速感知到设备离线停止发送指令。

retryInterval  会话内超时重传时间根据官方文档的描述说明设置为 0 ，依靠 TCP 的重传机制保证消息的可靠性。

### 传输数据使用protobuf如何设计数据帧结构

我采用**4 字节固定头部 + Protobuf 单入口消息结构**。头部用于快速路由、判断序列号幂等、版本兼容；

业务数据使用 protobuf 设计，使用 oneof 保存指令内容、告警信息或状态数据。

```
┌─────────────────────┬───────────────────────────┐
│    固定头部(4字节)    │     Protobuf 序列化载荷     │
├────────┬────────┬───┼───────────────────────────┤
│ 版本号  │ 消息类型 │ 序列号│  业务数据（protobuf）  │
│  1B    │   1B   │  2B  │     变长二进制          │
└────────┴────────┴───┴───────────────────────────┘
```

数据兼容性依靠 protobuf 的标准规范保证新旧终端和新旧网管的互通：

- 字段号分配后不修改、不复用。
- 新增字段需要用 optional 或者设置保留默认值，废弃字段标记 reserved 不删除。
- 通过版本号来进行兼容判断。

扩展性设计：

- 通过 oneof 来管理不同类型消息，所有消息类型都在一个 Message 结构中。
- 消息头中预留 4 字节拓展字段给未来使用。
- 通过消息头中的序列号来保证消息不会重复接收。
