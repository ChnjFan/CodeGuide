## 项目描述

这是一个全栈的即时通讯项目，移动端使用 flutter 通过 ClaudeCode 生成，主要编码实现了服务端。

服务端采用分布式设计，分为网关服务，多个聊天服务，状态服务和验证服务。

各服务之间通过 grpc 通信，支持断线重连，通过连接池来保证多服务并发访问。

网关对外采用 http 服务，负责处理用户登录和注册功能。网关服务在登录后向状态服务请求聊天服务的地址来实现负载均衡。

聊天服务采用 asio 库实现 tcp 可靠长连接的异步通信和消息转发，通过多线程封装 iocontext 的连接池来提升并发性能。

数据存储使用 mysql 服务，基于 mysqlconnector 库封装连接池，同时封装 redis 连接池出来缓存数据。

## 考察点

### 如何利用asio实现tcp服务

问我设计的聊天服务使用 asio 网络库，采用 Proactor 模式实现异步 IO。服务启动时就根据 CPU 的核数启动线程，io_context 作为事件循环核心，运行在每个独立的线程上。

注册回调处理异步完成事件，无阻塞 IO。

每个链接通过 session 管理，用智能指针管理 session 的生命周期，session 底层绑定的用户 uid，消息回调可以根据 session 查找用户进行消息推送。

收发数据时采用异步 async_read 和 async_write 方式读写，通过消息回调和消息队列完成数据收发。

数据统一用 json 格式，通过 TLV 方式对接收的 TCP 数据进行拆包防止粘包。通过心跳机制检测长连接的有效性。

详细的流程如下：

1、服务启动后根据 CPU 核数创建线程，每个线程独立运行 io_context。

2、初始化 acceptor，从连接池中获取 io_context 初始化一个 session，调用 async_accept 发起异步监听。

3、接收到连接后有 session 发起 async_read 异步读数据，等待客户端消息。

4、收到消息后，回调中根据 TLV 格式解析消息报文，根据消息 ID 调用处理函数，结束后将响应消息放入发送队列中，每次从发送队列取一条消息进行发送。

### IO 线程池和连接池的实现

IO 线程池专门运行 io_context::run()，处理所有的网络异步事件。

IO 线程池不直接管理连接，它管理的时连接上的网络事件。多个连接 socket 可以 IO 多路复用注册到同一个 io_context 上，IO 线程池通过轮询策略将连接均衡地注册到每个 io_context 上，避免单个线程连接过多，处理大量的网络事件。Linux 系统下 asio 网络库底层通过封装 epoll 来实现 IO 多路复用。

所有的网络操作都在 IO 线程中执行，服务启动后首先会从 IO 线程池中获取一个 io_context 初始化 session。调用异步 async_accept 监听到连接后，使用哈希表来维护 session，真正的连接管理是这个哈希表结构，通过心跳检测、超时清理来控制连接，保证服务稳定。

### 为什么要有心跳机制

TCP 连接发送 FIN 报文进行断连，这些在客户端主动断连或者网络不稳定的情况下发生，但是有些情况下客户端崩溃或者客户端直接拔网线操作不会主动发送 FIN 报文，对于服务端来说这些连接就变成了僵尸连接，占用服务器资源，长期运行下文件句柄资源耗尽后服务器无法再创建新连接。

### 如何保证服务高可用

通过状态服务来监控各聊天服务的健康状态，所有聊天服务定期向状态服务发送状态，由状态服务统一监控管理，如果发现某个聊天服务节点异常，对该节点流量进行隔离并做重启。

通过分布式的架构，将应用拆分为多个独立的服务，各服务可放在不同的数据中心，降低单个服务故障对整体系统的影响。

定期对数据备份，增加操作日志，服务故障后通过备份数据恢复和操作日志兜底补救。

### 为什么封装Mysql连接池

多个线程使用同一个mysql 连接存在线程安全问题，要为每个线程分配独立连接，而且连接数也不能无限增加，所以考虑设计连接池。

当每个线程想操作 Mysql 时从连接池取出连接进行数据访问。

连接池封装包括管理层和 Dao 数据访问层，管理层采用单例模式，Dao 层封装连接池，连接池使用队列实现生产者消费者模式管理可用连接，通过定时器访问 mysql 进行保活。

### Mysql 连接池如何处理连接断开和重连的问题

思路：**先检测有效性 → 发现失效就丢弃 → 自动重建新连接 → 保证业务无感知**。

1、借出前进行校验：发送测试 SQL：`SELECT 1` 测试连接有消息，响应正常则分配给业务，响应失败标记失效，重新建链

2、为了防止空闲连接超时断连，连接池创建后启动定时器每分钟（小于 Mysql 设置的wait_timeout）检测一遍连接是否正常。

### 连接池做过哪些性能和可靠性的考虑

性能方面：

1、配置最大连接数，常驻空闲连接应对突发流量，上限控制防止资源耗尽；

2、支持连接动态扩容和缩容，低峰时释放空闲连接，减少资源占用；

3、设置连接获取超时时间，避免线程长时间阻塞等待连接。

可靠性方面：

1、多级连接校验，借出前校验连接是否可用，定时巡检全部连接，剔除无效连接。

2、连接失效后重连机制，采用重试指数退避策略，防止频繁重试。

### 怎么保证多个服务节点的时钟同步问题

这个功能我还没有实现。

我了解到目前主流的实现方案是在机房内网上搭建 NTP（网络时间协议）服务集群，所有业务节点统一向内网源同步时间。

这样做相比公网 NTP 延迟更低、稳定性更强，局域网内能做到毫秒级误差，满足绝大多数业务需求。

对于金融实时交易系统这种高精度场景，使用 IEEE 1588 PTP 协议，依靠硬件时间戳实现微秒级同步。

### 性能测试

客户端初始多个线程定时间隔进行连接。

采用 pingpang 协议发送消息，单个服务节点连接消息收发稳定在 10ms 以内，不发生丢包和断连，增加连接数测试连接上线。

### 使用的设计模式和思想

- 单例模式实现连接管理和数据库管理类
- 生产者消费者模式，线程池内部使用队列管理线程，使用时从队列中取，结束后回收到队列中。
- RAII 思想，通过智能指针，封装 defer 从连接池获取连接后自动回收连接。
- protobuf 数据序列化压缩发送。
- TCP 发送数据使用发送队列保证异步顺序
