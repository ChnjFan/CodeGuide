# 分布式即时通讯项目

[TOC]

## 项目描述

这是一个全栈的即时通讯项目，移动端使用 flutter 通过 ClaudeCode 生成，主要编码实现了服务端。

服务端采用分布式设计，分为网关服务，多个聊天服务，状态服务和验证服务。

各服务之间通过 grpc 通信，支持断线重连，通过连接池来保证多服务并发访问。

网关对外采用 http 服务，负责处理用户登录和注册功能。网关服务在登录后向状态服务请求聊天服务的地址来实现负载均衡。

聊天服务采用 asio 库实现 tcp 可靠长连接的异步通信和消息转发，通过多线程封装 iocontext 的连接池来提升并发性能。

数据存储使用 mysql 服务，基于 mysqlconnector 库封装连接池，同时封装 redis 连接池出来缓存数据。

## 考察点

### 如何利用asio实现tcp服务

利用 asio 的多线程模式，根据 CPU 的核数来封装 io_context 的连接池，每个 io_context 跑在独立的线程上，采用异步 async_read 和 async_write 方式读写，通过消息回调和消息队列完成数据收发。

项目的网络模式是 Proactor 模式，每个连接通过 session 管理，通过 shared_ptr 管理 session，保证消息回调时 session 可用。底层绑定用户登录的 uid 和 session 相关联，消息回调可以根据 session 查找用户进行消息推送。

消息体统一用 json 格式，通过 TLV 方式对接收的 TCP 数据进行拆包防止粘包。通过心跳机制检测长连接的有效性。

### 为什么要有心跳机制

TCP 连接发送 FIN 报文进行断连，这些在客户端主动断连或者网络不稳定的情况下发生，但是有些情况下客户端崩溃或者客户端直接拔网线操作不会主动发送 FIN 报文，对于服务端来说这些连接就变成了僵尸连接，占用服务器资源，长期运行下文件句柄资源耗尽后服务器无法再创建新连接。

### 如何保证服务高可用

通过状态服务来监控各聊天服务的健康状态，所有聊天服务定期向状态服务发送状态，由状态服务统一监控管理，如果发现某个聊天服务节点异常，对该节点流量进行隔离并做重启。

通过分布式的架构，将应用拆分为多个独立的服务，各服务可放在不同的数据中心，降低单个服务故障对整体系统的影响。

定期对数据备份，增加操作日志，服务故障后通过备份数据恢复和操作日志兜底补救。

### 为什么封装Mysql连接池

多个线程使用同一个mysql 连接存在线程安全问题，要为每个线程分配独立连接，而且连接数也不能无限增加，所以考虑设计连接池。

当每个线程想操作 Mysql 时从连接池取出连接进行数据访问。

连接池封装包括管理层和 Dao 数据访问层，管理层采用单例模式，Dao 层封装连接池，连接池使用队列实现生产者消费者模式管理可用连接，通过定时器访问 mysql 进行保活。

### 性能测试

客户端初始多个线程定时间隔进行连接。

采用 pingpang 协议发送消息，单个服务节点连接消息收发稳定在 10ms 以内，不发生丢包和断连，增加连接数测试连接上线。

### 使用的设计模式和思想

- 单例模式实现连接管理和数据库管理类
- 生产者消费者模式，线程池内部使用队列管理线程，使用时从队列中取，结束后回收到队列中。
- RAII 思想，通过智能指针，封装 defer 从连接池获取连接后自动回收连接。
- protobuf 数据序列化压缩发送。
- TCP 发送数据使用发送队列保证异步顺序
