# MQTT 协议

MQTT（Message Queuing Telemetry Transport，消息队列遥测传输协议），是一种基于**发布/订阅（publish/subscribe）模式**的消息传输协议，该协议构建于 TCP/IP 协议上，可以以极少的代码和有限的带宽，为连接远程设备提供实时可靠的消息服务。

## 协议原理

主流的 MQTT 协议通过 TCP 连接，UDP 版本的叫做 MQTT-SN，两种版本基于不同连接方式优缺点也不同。

### 主要特性

MQTT 协议主要工作在低带宽、不可靠的网络的远程传感器和控制设备通讯而设计的协议。

- **轻量级：**开销低报文小（固定头部 2 字节），降低网络流量。
- **可靠传输：**使用 TCP/IP 提供网络连接，支持三种 QoS 等级、会话感知和持久连接。
- **安全通信：**提供传输层安全（TLS）和安全套接层（SSL）加密功能，通过用户名/密码凭证或客户端证书提供身份验证和授权机制。
- **双向通信：**发布/订阅模式提供双向通信方式。
- **连续、有状态的会话：**提供客户端与 Broker 之间保持有状态会话能力，保证通信可靠性，降低因间断性连接导致数据丢失的风险。

### 工作原理

MQTT 是基于发布-订阅模式的通信协议，客户端通过主题 Topic 发布或订阅消息，通过 Broker 集中管理消息路由，根据预设 QoS 确保消息传递可靠性。

- 发布者（Publisher）：消息发送方，只需要将消息发布到指定的主题（Topic）上，也可以是订阅者。
- 订阅者（Subscriber）：消息接收方，订阅消息 Topic 后服务端会主动推送消息。
- 服务端（Broker）：消息代理，接收发布者消息，管理 Topic 和客户端订阅关系，将消息推送给订阅者，同时还负责所有客户端的连接认证、会话管理和离线消息存储。

消息传输包含主题 Topic 和负载 payload：

- Topic：消息的类型，订阅者订阅主题后收到该主题的消息。
- payload：消息内容。

### 发布-订阅模式

发布-订阅模式通过 Broker 将发送消息的发布者和接收消息的订阅者解耦，发布者与订阅者之间无需建立直接连接，而是通过 Broker 负责消息路由和分发。

在 MQTT 中主题和订阅无法提前注册或创建，Broker 无法预知主题会有多少个订阅者，只能将消息转发给当前的订阅者，如果不存在订阅者消息会被丢弃。

客户端既可以是发布者，也可以是订阅者。

### 主题 Topic

MQTT 主题是消息的寻址方式，本质是一个 UTF-8 编码的字符串，通过 `/` 区分层级：

```
chat/room/1
sensor/10/temperature
sensor/+/temperature
```

- 通常主题不以 `/` 开头或结尾，例如 `/chat` 或 `chat/`。
- 客户端在订阅或发布时自动创建主题，不需要手动创建或删除。
- 协议规定长度为两个字节，最长包含 65535 个字符。
- 不同 Broker 对最大主题数量支持不同，越多会消耗更多的服务器内存，建议一个客户端订阅主题数量不超过 10 个。

#### 通配符

通配符 `+` 和 `#` 用于客户端一次订阅多个主题，**只能用于客户端订阅**，必须占据整个层级。

- `+`：表示单层通配符。
- `#`：表示多层通配符，必须是主题的最后一个字符。

#### 系统主题

以 `$SYS/` 开头的主题为系统主题，用于获取 Broker 运行状态等数据。

#### 共享订阅

共享订阅是 MQTT 5.0 引入的新特性，以 `$share/` 开头，用在多个订阅者直接订阅的负载均衡。

### QoS

MQTT QoS 指发布者与订阅者之间消息传递的可靠级别，提供三个服务级别：

- QoS 0：最多交付一次，可能会丢失消息。
- QoS 1：至少交付一次，可以保证收到消息，可能会有重复。
- QoS 2：只交付一次，可以保证消息既不丢失，也不会重复。

QoS 等级由发布者在 PUBLISH 报文中指定，Broker 在转发时可根据订阅者的订阅要求将消息的 QoS 等级发生降级转发。

#### QoS 0 - 最多交付一次

消息发送后即可结束，不需要等待确认和重传。消息丢失的频率依赖网络环境，可能会错过断开连接期间的消息。

使用 QoS 0 传输高频但不重要的数据，比如传感器数据，周期更新可以接收丢失部分数据。

**为什么 QoS 0 消息会丢失？**

MQTT QoS 0 传递消息，消息的可靠性完全依赖 TCP 协议。TCP 协议只能保证在连接稳定不关闭的情况下消息可靠传输，如果发生连接关闭、重置情况，正处于网络链路或系统底层缓冲区的消息将会丢失。

#### QoS 1 - 至少交付一次

QoS 1 加入应答与重传机制，发送方只有接收到 PUBACK 报文才认为消息传递成功。

PUBLISH 报文设置 PacketID 作为 PUBACK 报文对应的标记。

适合传输重要的数据，如下达关键指令，更新重要有实时性的状态等。

协议层无法避免消息重复，需要在业务层通过**时间戳或者消息流水号**来对消息进行去重。

**为什么 QoS 1 消息会重复？**

发送方未收到 PUBACK 报文后会重发消息，存在两种情况：

1. PUBLISH 未到达接收方，此时重传后不会收到重复消息。
2. PUBLISH 已经到达接收方，发送方未收到 PUBACK 报文，此时重发报文后接收方会收到重复消息。

虽然重传消息设置了 DUP 标志，但是 PacketID 可能会复用，因此接收方不能通过 PacketID + DUP 认为此次消息是重复接收的消息。

#### QoS 2 - 只交付一次

QoS 2 确保接收方只收到一次消息，新增 PUBREL 报文和 PUBCOMP 报文流程保证消息不会重复。

1. 发送方存储并发送 PUBLISH 报文，等待接收方回复 PUBREC 报文。
2. 发送方收到 PUBREC 报文则不需要重传报文，发送方删除本地存储 PUBLISH 报文，然后存储并发送 PUBREL 报文通知对端 PacketID 可使用。
3. 接收方收到 PUBREL 报文后，回复 PUBCOMP 报文表示自己准备好接收该 PacketID 用于新消息。
4. 发送方收到 PUBCOMP 报文后消息传输正式完成，之后发送方可再次使用 PacketID 发送新消息。

**为什么 QoS 2 消息不会重复？**

QoS 1 消息的接收方在发送完 PUBACK 报文后就认为 PacketID 可以重新使用，不管 PUBACK 报文有没有到达发送方，所以无法得知收到的相同 PacketID 的 PUBLISH 报文是新报文还是重传报文。

QoS 2 增加 PUBREL 流程用来协商 PacketID 是否可以重用。

- 发送方在收到 PUBREC 报文前可以重传 PUBLISH 报文，在收到 PUBREC 报文后就进入 PacketID 释放流程，不能再使用 PacketID 重传 PUBLISH 报文。
- 收到对端回复的 PUBCOMP 报文确认双方 PacketID 释放之前，不可使用当前 PacketID 发送新消息。

对于接收方，收到 PUBREL 报文之前，相同 PacketID 的消息都是重复消息，发送 PUBCOMP 报文后收到的消息都是新消息。

## 工作流程

MQTT 客户端使用 TCP/IP  协议与 Broker 建立连接，可选择使用 TLS/SSL 加密实现安全通信。

客户端提供认证信息，并指定会话类型。

客户端向特定主题发布消息，也可订阅主题接收消息。

- 客户端发布消息，将消息发送给 Broker。
- 客户端订阅消息，接受与订阅主题相关的消息。
- Broker 接收发布的消息，将消息转发给订阅主题的客户端，根据 QoS 等级确保消息可靠传递，根据会话类型为断开连接的客户端存储消息。
