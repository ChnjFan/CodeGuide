# MQTT 协议

MQTT（Message Queuing Telemetry Transport，消息队列遥测传输协议），是一种基于**发布/订阅（publish/subscribe）模式**的消息传输协议，该协议构建于 TCP/IP 协议上，可以以极少的代码和有限的带宽，为连接远程设备提供实时可靠的消息服务。

## 协议原理

主流的 MQTT 协议通过 TCP 连接，UDP 版本的叫做 MQTT-SN，两种版本基于不同连接方式优缺点也不同。

### 主要特性

MQTT 协议主要工作在低带宽、不可靠的网络的远程传感器和控制设备通讯而设计的协议。

- **轻量级：**开销低报文小（固定头部 2 字节），降低网络流量。
- **可靠传输：**使用 TCP/IP 提供网络连接，支持三种 QoS 等级、会话感知和持久连接。
- **安全通信：**提供传输层安全（TLS）和安全套接层（SSL）加密功能，通过用户名/密码凭证或客户端证书提供身份验证和授权机制。
- **双向通信：**发布/订阅模式提供双向通信方式。
- **连续、有状态的会话：**提供客户端与 Broker 之间保持有状态会话能力，保证通信可靠性，降低因间断性连接导致数据丢失的风险。

### 工作原理

MQTT 是基于发布-订阅模式的通信协议，客户端通过主题 Topic 发布或订阅消息，通过 Broker 集中管理消息路由，根据预设 QoS 确保消息传递可靠性。

- 发布者（Publisher）：消息发送方，只需要将消息发布到指定的主题（Topic）上，也可以是订阅者。
- 订阅者（Subscriber）：消息接收方，订阅消息 Topic 后服务端会主动推送消息。
- 服务端（Broker）：消息代理，接收发布者消息，管理 Topic 和客户端订阅关系，将消息推送给订阅者，同时还负责所有客户端的连接认证、会话管理和离线消息存储。

消息传输包含主题 Topic 和负载 payload：

- Topic：消息的类型，订阅者订阅主题后收到该主题的消息。
- payload：消息内容。

### 发布-订阅模式

发布-订阅模式通过 Broker 将发送消息的发布者和接收消息的订阅者解耦，发布者与订阅者之间无需建立直接连接，而是通过 Broker 负责消息路由和分发。

在 MQTT 中主题和订阅无法提前注册或创建，Broker 无法预知主题会有多少个订阅者，只能将消息转发给当前的订阅者，如果不存在订阅者消息会被丢弃。

客户端既可以是发布者，也可以是订阅者。

### 主题 Topic

MQTT 主题是消息的寻址方式，本质是一个 UTF-8 编码的字符串，通过 `/` 区分层级：

```
chat/room/1
sensor/10/temperature
sensor/+/temperature
```

- 通常主题不以 `/` 开头或结尾，例如 `/chat` 或 `chat/`。
- 客户端在订阅或发布时自动创建主题，不需要手动创建或删除。
- 协议规定长度为两个字节，最长包含 65535 个字符。
- 不同 Broker 对最大主题数量支持不同，越多会消耗更多的服务器内存，建议一个客户端订阅主题数量不超过 10 个。

#### 通配符

通配符 `+` 和 `#` 用于客户端一次订阅多个主题，**只能用于客户端订阅**，必须占据整个层级。

- `+`：表示单层通配符。
- `#`：表示多层通配符，必须是主题的最后一个字符。

#### 系统主题

以 `$SYS/` 开头的主题为系统主题，用于获取 Broker 运行状态等数据。

#### 共享订阅

共享订阅是 MQTT 5.0 引入的新特性，以 `$share/` 开头，用在多个订阅者直接订阅的负载均衡。

### QoS

MQTT QoS 指发布者与订阅者之间消息传递的可靠级别，提供三个服务级别：

- QoS 0：最多交付一次，可能会丢失消息。
- QoS 1：至少交付一次，可以保证收到消息，可能会有重复。
- QoS 2：只交付一次，可以保证消息既不丢失，也不会重复。

QoS 等级由发布者在 PUBLISH 报文中指定，Broker 在转发时可根据订阅者的订阅要求将消息的 QoS 等级发生降级转发。

#### QoS 0 - 最多交付一次

消息发送后即可结束，不需要等待确认和重传。消息丢失的频率依赖网络环境，可能会错过断开连接期间的消息。

使用 QoS 0 传输高频但不重要的数据，比如传感器数据，周期更新可以接收丢失部分数据。

**为什么 QoS 0 消息会丢失？**

MQTT QoS 0 传递消息，消息的可靠性完全依赖 TCP 协议。TCP 协议只能保证在连接稳定不关闭的情况下消息可靠传输，如果发生连接关闭、重置情况，正处于网络链路或系统底层缓冲区的消息将会丢失。

#### QoS 1 - 至少交付一次

QoS 1 加入应答与重传机制，发送方只有接收到 PUBACK 报文才认为消息传递成功。

PUBLISH 报文设置 PacketID 作为 PUBACK 报文对应的标记。

适合传输重要的数据，如下达关键指令，更新重要有实时性的状态等。

协议层无法避免消息重复，需要在业务层通过**时间戳或者消息流水号**来对消息进行去重。

**为什么 QoS 1 消息会重复？**

发送方未收到 PUBACK 报文后会重发消息，存在两种情况：

1. PUBLISH 未到达接收方，此时重传后不会收到重复消息。
2. PUBLISH 已经到达接收方，发送方未收到 PUBACK 报文，此时重发报文后接收方会收到重复消息。

虽然重传消息设置了 DUP 标志，但是 PacketID 可能会复用，因此接收方不能通过 PacketID + DUP 认为此次消息是重复接收的消息。

#### QoS 2 - 只交付一次

QoS 2 确保接收方只收到一次消息，新增 PUBREL 报文和 PUBCOMP 报文流程保证消息不会重复。

1. 发送方存储并发送 PUBLISH 报文，等待接收方回复 PUBREC 报文。
2. 发送方收到 PUBREC 报文则不需要重传报文，发送方删除本地存储 PUBLISH 报文，然后存储并发送 PUBREL 报文通知对端 PacketID 可使用。
3. 接收方收到 PUBREL 报文后，回复 PUBCOMP 报文表示自己准备好接收该 PacketID 用于新消息。
4. 发送方收到 PUBCOMP 报文后消息传输正式完成，之后发送方可再次使用 PacketID 发送新消息。

**为什么 QoS 2 消息不会重复？**

QoS 1 消息的接收方在发送完 PUBACK 报文后就认为 PacketID 可以重新使用，不管 PUBACK 报文有没有到达发送方，所以无法得知收到的相同 PacketID 的 PUBLISH 报文是新报文还是重传报文。

QoS 2 增加 PUBREL 流程用来协商 PacketID 是否可以重用。

- 发送方在收到 PUBREC 报文前可以重传 PUBLISH 报文，在收到 PUBREC 报文后就进入 PacketID 释放流程，不能再使用 PacketID 重传 PUBLISH 报文。
- 收到对端回复的 PUBCOMP 报文确认双方 PacketID 释放之前，不可使用当前 PacketID 发送新消息。

对于接收方，收到 PUBREL 报文之前，相同 PacketID 的消息都是重复消息，发送 PUBCOMP 报文后收到的消息都是新消息。

## 工作流程

MQTT 客户端使用 TCP/IP  协议与 Broker 建立连接，可选择使用 TLS/SSL 加密实现安全通信。

客户端提供认证信息，并指定会话类型。

客户端向特定主题发布消息，也可订阅主题接收消息。

- 客户端发布消息，将消息发送给 Broker。
- 客户端订阅消息，接受与订阅主题相关的消息。
- Broker 接收发布的消息，将消息转发给订阅主题的客户端，根据 QoS 等级确保消息可靠传递，根据会话类型为断开连接的客户端存储消息。

### 客户端连接参数

MQTT 连接由客户端向服务端发起，首先发送一个 CONNECT 报文给服务端，服务端回复 CONNACK 后连接建立成功。客户端超时时间内未收到 CONNACK 报文就主动关闭连接。

#### 连接地址

MQTT 连接地址包含：服务器 IP 或域名、端口号、连接协议。

- 普通 TCP 连接：`mqtt://broker.io:1883`
- 基于 TLS/SSL 的 TCP 安全连接：`mqtts://broker.io:8883`
- WebSocket 连接：`ws://broker.io:8083`
- WebSocket 安全连接：`wss://broker.io:8084`

### 会话

MQTT 会话存储服务端与客户端额外的上下文数据，网络中断恢复后能够恢复通信。通过会话 QoS 1 和 QoS 2 消息的服务质量才能得到保障。

服务端通过 Client ID 唯一标识每个会话，客户端连接复用之前的会话，必须使用相同的 Client ID。

#### Clean Start

MQTT 5.0 在 CONNECT 报文提供 Clean Start 字段指定是否复用已存在的会话。

- 值为 0 如果服务端存在指定 Client ID 关联会话，必须使用该会话恢复通信。
- 值为 1 服务端必须丢弃已存在的会话，重新建立新的会话。

#### Session Expiry Interval

MQTT 5.0 在 CONNECT 报文提供 Session Expiry Interval 字段指定会话在网络中断后的保留时间。

每个客户端可独立设置，再断开连接时，可以在 DISCONNECT 报文中设置该属性清除保留时间。

客户端在上线时设置一个大于 0 的值，避免网络中断影响正常业务，客户端完成所有业务主动下线时，将会话过期时间更新为 0，服务端及时释放会话。

#### Clean Session

MQTT 3.1.1 会话机制只有 Clean Session 字段。

- 值为 0 时复用会话，并将会话设置为永不过期。
- 值为 1 时新建会话，且会话生命周期与网络连接一致。

## 特殊消息

### 保留消息

发布消息设置 Retained 标记为 true，服务器会为每个主题存储最新一条交流消息，客户端在之后上线订阅主题时可以接收到保留消息。

订阅者无法主动向发布者请求消息，订阅者收消息完全依赖发布者何时发布消息。保留消息让订阅者立刻获得最新状态，不需要等待。

### 遗嘱消息

Will Message 让客户端知道其他客户端是否在线，让意外离线的客户端优雅处理结束流程。

- 客户端在连接时向服务端注册遗嘱消息，设置遗嘱消息的主题、有效载荷等。
- 客户端意外断开连接，服务端就向订阅主题的客户端发送遗嘱消息。

遗嘱消息只有在客户端以外断开连接的情况下才会发送，只要网络连接在服务端没有收到 Reason Code 为 0x00 的 DISCONNECT 报文情况下关闭，服务端都会发送遗嘱消息。

#### Will Delay Interval

MQTT 5.0 增加 Will Delay Interval 属性用来延迟发布遗嘱消息，服务端在网络连接关闭后延迟发布遗嘱消息，如果在这期间客户端恢复连接，那么遗嘱消息将不会发布。

遗嘱消息是服务端会话状态中的一部分，会话结束服务端也必须发布遗嘱消息。

MQTT 3.1.1 的遗嘱消息在网络连接关闭时立即发布。

#### 使用场景

1. 将遗嘱消息设置为保留消息，客户端上线后订阅消息仍可接收到遗嘱消息，感知客户端的在线状态。
2. 会话过期通知，将延迟时间设置比会话超期时间大，这样会话超期时会发布遗嘱消息给客户端。

及时主动下线，客户端也可以发送一个 Reason Code 不为 0x00 的 DISCONNECT 报文要求服务端发送遗嘱消息。

### 请求响应机制

MQTT 5.0 之前版本的请求响应机制，通过互相协商好请求和响应的主题，然后订阅端在收到请求后向响应主题返回响应。

当存在多个不同请求方，所有请求方只能订阅相同响应主题，所有请求方都会收到响应，无法分清响应是否属于自己。

#### 响应主题

MQTT 5.0 请求方在请求消息中指定响应主题（Response Topic），响应方向请求中的响应主题发布响应消息。

请求方将自己的 Client ID 作为响应主题的一部分，避免不同请求方使用相同的响应主题造成冲突。

#### 关联数据

请求消息携带关联数据（Correlation Data），响应方必须在响应中将关联数据返回，请求方根据关联数据识别响应所属的原始请求。

- 响应方没有按请求顺序返回响应，或者网络断连丢失响应，请求方根据关联数据识别响应避免请求方不正确关联响应和原始请求。
- 请求方仅订阅一个响应主题管理多个响应方异步返回的响应。

#### 响应信息

服务端通常会限制客户端发布和订阅主题，请求方指定的响应主题无法保证有权限订阅。

响应消息属性通过 CONNECT 报文中将请求响应信息标识符设置为 1，服务端在 CONNACK 报文中返回响应信息。

客户端将响应信息内容作为响应主题的特定部分，通过服务端的权限检查。

#### 最佳实践

1. QoS 1 和 QoS 2 只能确保消息到达服务端，请求响应机制确认消息到达订阅端。
2. 发送消息前先订阅响应主题，以免错过响应。
3. 响应消息帮助构建符合权限要求的响应主题。
4. 请求方使用 Client ID 作为响应主题一部分，避免多个请求方时响应消息混淆。
5. 请求方设置关联数据，避免多个响应方返回的响应消息混淆。
6. 遗嘱消息使用请求响应机制，需要连接时为遗嘱消息设置响应主题，客户端可以知道离线期间遗嘱消息是否被消费。

## 消息属性

### 用户属性

用户属性（User Properties）作为自定义属性，允许用户在 MQTT 消息中添加元数据扩展应用场景。

- UTF-8 的键值对数组，消息属性字段配置，不超过最大消息大小。
- 客户端与 Broker 发起连接时，服务器预设用户属性，客户端填写后由服务器使用。
- 消息发布的用户属性在客户端之间进行传递，如消息编号、时间戳、文件、客户端信息和路由信息等。

使用场景：

- 文件传输：用户属性保存文件名等信息，Payload 直接保存文件内容。
- 资源解析：用户属性保存资源类型和解析方式，Payload 保存资源内容。
- 消息路由：用户属性保存消息类型和时间戳等，Payload 保存消息体。

### 主题别名

问题场景：发布大量相同主题消息，每条消息都写入相同主题名，浪费带宽资源和解析时的计算资源。

主题别名（Topic Alias）将主题长度较长且常用的主题名缩减为**两字节整数**降低发布消息的带宽消耗。

- 属性字段中，双字节整数。
- 编码在 PUBLISH 报文可变报头部分，值 `0x23` 标识后两个字节是主题别名。
- CONNECT 报文和 CONNACK 报文交换主题别名最大长度。
- 主题别名不能是 0，也不允许超过主题别名最大值。

生命周期和作用范围：仅限于当前连接，连接断开重连后需要重新建立主题别名和主题名的映射关系。

### Payload Format Indicator

Payload Format Indicator 是 MQTT 5.0 引入的属性用来指示报文中有效载荷的格式。

PUBLISH 报文和 CONNECT 报文中的遗嘱消息需要声明格式，其他报文格式都是固定的。

- 值为 0 时 Payload 是未指定的字节流
- 值为 1 时 Payload 是 UTF-8 编码字符数据，接收者先校验数据格式是否正确。

### Content Type

Content Type 的值是一个 UTF-8 编码的字符串，用来描述应用消息的内容，帮助接收端如何解析有效载荷。

具体内容由发送端和接收端决定。

### 共享订阅

共享订阅让 MQTT 服务端使用特定订阅的客户端之间均衡分配消息负载，当有多个客户端共享一个订阅时，每个匹配该订阅的消息都只会有一个副本传递给一个客户端。

共享订阅主题命名规范：

```
$share/{Share Name}/{Topic Filter}
```

`$share` 固定前缀，`{Share Name}` 客户端指定标识共享订阅使用的共享名，`{Topic Filter}` 实际订阅的主题。

使用相同共享名组成一组订阅会话，服务端向匹配的订阅会话中的其中一个客户端发送消息副本。

### Keep Alive

为了避免 TCP 协议出现半链接问题导致的通信黑洞，MQTT 提供 Keep Alive 机制，让客户端和服务器判定是否存在半链接问题从而关闭连接。

客户端创建连接时设置 Keep Alive 可变头部字段为 0~65535，表示客户端两次协议包之间的最大间隔时间。

1. 客户端连接建立后，需要在 Keep Alive 时间间隔内发送消息。空闲状态可发送 PINGREQ 协议包，服务端回复 PINGRESP 协议包。
2. Broker 在 Keep Alive 的 1.5 倍时间内没有收到客户端的协议包，认为连接出现问题而断开连接。
3. Broker 存在半连接时，客户端发起重连或新建连接时 Broker 启动客户端接管机制，关闭旧的半连接，然后与客户端建立新的连接。

### Message Expiry Interval

消息过期间隔允许发布端为有时效性的消息设置过期间隔，消息在服务端超过时间间隔后就不会再分发给订阅者。

在持久会话中可以为离线客户端缓存未发送的消息，但是客户端如果离线时间较长就没有必要再发送不重要的消息。
